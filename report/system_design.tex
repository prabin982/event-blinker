\chapter{System Design}

\section{System Architecture}
The architecture of \textbf{Event Blinker} follows a three-tier client-server model with real-time communication:

\begin{itemize}[noitemsep]
    \item \textbf{Presentation Layer:} A multi-client ecosystem comprising a cross-platform React Native (Expo) mobile application, a responsive React/Vite web portal for event organizers, and a lightweight standalone HTML/JS admin portal. This decoupling allows each interface to be optimized for its specific user base.
    \item \textbf{Application Layer:} A centralized Node.js/Express.js API server that handles business logic, authentication, and integration with third-party services. The Socket.io engine is embedded within this layer to facilitate bidirectional, real-time message broadcasting for chat and ride updates.
    \item \textbf{Data Layer:} A relational PostgreSQL database with the PostGIS extension for high-performance spatial queries, complemented by a local file storage system for managing document and image uploads.
\end{itemize}

By adopting this modular architecture, \textbf{Event Blinker} ensures that frontend clients can evolve independently of the backend logic. The three-tier model also enhances security by isolating the database from direct external access, forcing all requests to pass through the authentication and validation middleware in the application layer.

\section{Database Design}

\subsection{Entity-Relationship Overview}
The database consists of 12+ tables organized into three domains: Users, Events, and Rides. Each domain is interlinked using foreign key constraints to maintain referential integrity. All primary keys are stored as Universally Unique Identifiers (UUIDs) to ensure data portability across distributed instances and to prevent easy enumeration of resource IDs. Furthermore, the use of PostGIS's geometry types allows for native spatial indexing via GiST, which is absolutely critical for the platform's ability to filter thousands of events based on a user's current GPS coordinates in milliseconds.

\section{System Diagrams}

\subsection{Use Case Diagram}
The use case diagram identifies four primary actors and their interactions with the system:
\begin{itemize}[noitemsep]
    \item \textbf{User/Attendee:} Discover events on map, view details, like, check-in, chat, book rides, interact with Blinker AI.
    \item \textbf{Event Organizer:} Create events, manage attendees, view analytics, moderate chat.
    \item \textbf{Driver/Rider:} Register with vehicle/license, go online, accept ride requests, offer custom prices.
    \item \textbf{Admin:} Approve/reject organizers, events, riders, and licenses.
\end{itemize}
